## Current Card Types

### Article Card

- Template id: `plick_transfer_v1`
- Admin mode id: `article`
- Admin UI:
  - `CardNewsWorkspace`
  - `CardNewsPreview`
  - `CardPreviewPage`
- Render server schema:
  - `CardNewsContent`
  - `CardCover`
  - `CardDetail`
- Render template directory:
  - `app/render/templates/plick_transfer_v1`
- Expected render request shape:

```json
{
  "template_id": "plick_transfer_v1",
  "image_url": "https://...",
  "card": {
    "cover": {
      "subject": "LIV",
      "subject_color": "#FB0009",
      "headline": "헤드라인",
      "summary": "요약"
    },
    "detail": {
      "paragraphs": "본문 1\n\n본문 2",
      "source": "Fabrizio Romano"
    }
  }
}
```

Admin behavior:

- User selects a published item and optionally uploads or enters an image URL.
- AI draft can populate subject/headline/summary/detail fields.
- `cardFromFields()` converts UI fields into `card`.
- `publishCardNews()` creates a publication with `kind: "article"` and `template_id: "plick_transfer_v1"`.
- Local preview uses React/CSS only; final output is generated by the card render server.

Render behavior:

- `RenderRequest` requires both `image_url` and `card` for `plick_transfer_v1`.
- `build_render_html_pages_from_parts()` creates page 1 cover HTML and page 2 detail HTML.
- `render_pages_to_files()` renders PNGs through Playwright.
- `write_cardnews_zip_file()` writes rendered pages and appends `outtro.png`.

- Template id: `plick_today_fixtures_v1`
- Admin schedule type: `today`
- Admin UI:
  - `TodayFixturesEditor`

- Render template directory:

- Background asset:

Expected render request shape:

```json
{
  "template_id": "plick_today_fixtures_v1",
  "today_fixtures": {
    "schedule_type": "today",
    "eyebrow": "WORLD CUP 2026 · TODAY",
    "title": "오늘의\n경기 일정",
    "date_label": "6월 12일 (금)",
    "matches": [
      {
        "time_period": "오전",
        "kickoff_time": "04:00",
        "home_team": "브라질",
        "home_code": "BRA",
        "home_image_url": "https://...",
        "away_team": "세네갈",
        "away_code": "SEN",
        "away_image_url": "https://...",
        "group_label": "Group F",
        "venue": "마이애미 · 하드록 스타디움"
      }
    ]
  }
}
```

Admin behavior:

- `TODAY_FIXTURES_MATCHES_PER_PAGE = 4`.
- `TODAY_FIXTURES_MAX_MATCHES = 40`.
- Presets control eyebrow base and group options:
  - World Cup: `WORLD CUP 2026`, `Group A-H`
  - EPL: `PREMIER LEAGUE`, `Matchweek 1-38`
  - Champions League: `CHAMPIONS LEAGUE`, round labels
- `todayFixturesPayload()` normalizes country codes, team names, image URLs, group label, and venue.
- `todayFixturesValidationIssues()` requires:
  - date label
  - 1-40 matches
  - kickoff time
  - home country
  - away country
  - group/round
  - venue
- `TodayFixturesPreview` chunks flat `matches` into 4-match pages.
- Preview page buttons are local preview state only; generated PNG page navigation is handled separately in `GeneratedCardNewsArchive`.

Render behavior:

- `TodayFixturesContent` carries `matches` as the canonical flat list.
- `build_today_fixtures_html_pages()` splits every 4 matches.
- `build_today_fixtures_html()` renders each page with `match_rows`.
- `_today_fixture_match_to_html()` renders time, home/away team, country badges, `VS`, and meta line (`group_label · venue`).
- Long team names are scaled with `_fixture_team_font_size()`.

Admin behavior:

- Weekly uses the same match fields as today for core data:
  - period
  - kickoff time
  - home country
  - away country
- Weekly removes group/round and venue from the UI/validation.
- `days` is the weekly-specific grouped source of truth in the UI.
- `matches` is still sent as a flattened compatibility list.
- `weeklyFixtureDaysForValue()` converts existing flat matches into one day when switching from today to weekly.
- `flattenWeeklyFixtureDays()` keeps the compatibility `matches` list synced.
- `buildWeeklyFixturePreviewPages()` paginates by date group:
  - target maximum: 6 matches per page
  - keep a date together when possible
  - if a single date has more than 6 matches, split that date and repeat the date header on the next page
- `TodayFixturesPreview` has a weekly branch with smaller typography based on the Figma reference:
  - eyebrow around top 86
  - title around top 146
  - range around top 346
  - first date header around top 425
  - match rows step by 106px
- A preview page button should only change preview page state. Do not let selected-match auto-scroll force the page back unless the selected match actually changed.

Render behavior:

- `TodayFixturesContent` accepts `schedule_type`, `matches`, and optional `days`.
- `TodayFixtureDay` validates `date`, `date_label`, and grouped `matches`.
- `build_weekly_fixtures_html_pages()` uses `days` when present.
- If `days` is missing, it falls back to one day built from `date_label` and flat `matches`.
- `_weekly_fixture_match_to_html()` renders only:
  - time period
  - kickoff time
  - home team
  - home country badge
  - VS
  - away country badge
  - away team
- It intentionally does not render `group_label` or `venue`.
- Weekly long team names use `_weekly_fixture_team_font_size()` with a smaller base size than today.

## Admin Publication Flow

### Immediate ZIP Download

- UI function:
  - `renderTodayFixtures()`
- API route:
  - `POST /api/admin/card-template-render`
- Use case:
  - Generate and download a fixtures ZIP directly.
- Route behavior:
  - Requires admin token.
  - Accepts `plick_today_fixtures_v1`, `plick_weekly_fixtures_v1`, `plick_today_results_v1`, and `plick_weekly_results_v1`.
  - Forwards to card render server `POST /card/render`.
  - Returns ZIP bytes.
- Filename prefix:
  - today: `cardnews-today-fixtures-*`
  - weekly: `cardnews-weekly-fixtures-*`
  - today results: `cardnews-today-results-*`
  - weekly results: `cardnews-weekly-results-*`

### R2 Upload / Publication

- UI functions:
  - `publishCardNews()`
  - `publishTodayFixtures()`
  - `startCardPublication()`
- API routes:
  - `GET /api/admin/card-publications`
  - `POST /api/admin/card-publications`
  - `DELETE /api/admin/card-publications`
  - `POST /api/admin/card-publications-sync`
- Supabase table:
  - `card_news_publications`
- Publication kinds:
  - `article`
  - `today_fixtures`
- Template ids are stored separately in `template_id`, so weekly fixtures and match results still use `kind: "today_fixtures"` with their specific fixture/result template id.
- `source_payload` stores the original source data needed for recovery/requeue.
- `render_request` is sent to the card render server `POST /card/render-jobs`.
- `card-publications-sync` polls `GET /card/render-jobs/{job_id}` and patches:
  - status
  - pages
  - zip_url
  - r2_prefix
  - error_message
  - render timings
- If a render job is missing, sync can requeue once using `source_payload`.
- For fixture requeue, sync must preserve weekly template id; do not hard-code all fixture requeues to `plick_today_fixtures_v1`.

## Generated Card News Archive

- Admin component:
  - `GeneratedCardNewsArchive`
- Purpose:
  - Display previously generated cardnews publications.
  - Navigate between generated cards.
  - Navigate between PNG pages in the selected card.
  - Download ZIP or open the current image.
  - Delete generated publication and storage objects.
- Important state:
  - Selected publication is tracked by `activePublicationId`, not array index.
  - Page inside selected publication is tracked by `pageIndex`.
- Do not confuse this with `TodayFixturesPreview`:
  - `GeneratedCardNewsArchive` previews completed/generated PNGs from R2.
  - `TodayFixturesPreview` is a local live preview before rendering.

## Card Render Server Flow

Main endpoints in `app/api/card.py`:

- `POST /card/draft`
  - Builds article card draft JSON from summary input.
- `POST /card/render`
  - Synchronous ZIP render.
  - Uses `create_render_zip_file()`.
  - Writes rendered PNG pages plus outtro.
- `POST /card/render-jobs`
  - Asynchronous render and upload job.
  - Used by admin publication flow.
- `GET /card/render-jobs/{job_id}`
  - Poll current job status.
- `POST /card/storage/delete`
  - Deletes R2 keys for generated pages/ZIP.
- `POST /card/render-upload`
  - Legacy/manual upload flow for article cards with uploaded image file.

Core render functions in `app/services/renderer.py`:

- `create_render_zip()`
  - In-memory ZIP with manifest.
  - Used by tests and direct service calls.
- `create_render_zip_file()`
  - Writes a ZIP file from rendered page files.
  - Used by API endpoints and render jobs.
- `create_render_zip_from_parts()`
  - Article card in-memory ZIP.
- `create_today_fixtures_zip()`
  - Today fixtures in-memory ZIP.
- `create_weekly_fixtures_zip()`
  - Weekly fixtures in-memory ZIP.
- `build_render_html_pages_from_parts()`
  - Article page HTML.
- `build_today_fixtures_html_pages()`
  - Today fixtures pagination.
- `build_weekly_fixtures_html_pages()`
  - Weekly fixtures pagination.
- `render_pages_to_files()`
  - Playwright HTML-to-PNG rendering.
- `write_cardnews_zip_file()`
  - Packs pages and outtro into ZIP for API/job flow.

Async job flow in `app/services/render_jobs.py`:

- `submit_render_job()` validates R2 config, creates in-memory job record, then starts `_run_render_job()`.
- `_html_pages_for_request()` dispatches by template id:
  - article -> materialize article image URL, then article HTML pages
  - today fixtures -> today pages
  - weekly fixtures -> weekly pages
- `_run_render_job()` renders pages, uploads PNGs to R2, then writes/uploads ZIP.
- Job status values:
  - `queued`
  - `running`
  - `zip_pending`
  - `completed`
  - `failed`

## Template And Layout Notes

- All cards are 1080x1350, 4:5.
- Shared preview constants in admin:
  - `CARD_PREVIEW_WIDTH = 1080`
  - `CARD_PREVIEW_HEIGHT = 1350`
  - `CARD_PREVIEW_MAX_WIDTH = 360`

- Today fixtures:
  - 4 matches per page
  - larger row typography
  - includes meta line
- Weekly fixtures:
  - 6 matches per page target
  - smaller row typography
  - date headers and divider lines
  - no match card boxes
  - no group/venue meta
- Country badges:
  - Admin: `FixtureFlagBadge`

  - Missing image URL falls back to country code inside dashed circle.
- Some country display names are shortened for cards:
  - Admin uses `getTodayFixtureCountryCardName()`
  - Render server uses `_fixture_team_card_name()`

## Adding A New Card Type

When adding a new card type, update both sides deliberately.

Admin repo checklist:

- Add template id constant in `src/epl/AdminDashboard.jsx`.
- Add UI mode if the new card needs its own editor.
- Add payload builder and validation.
- Add local live preview if the user needs visual feedback before rendering.
- Decide whether the card is immediate ZIP only, publication/R2 upload, or both.
- If using publication flow, preserve `template_id` in `card_news_publications`.
- Update API handlers if they need to allow the new template id.
- Update generated archive only if page/image navigation behavior changes.

Card render repo checklist:

- Add enum value to `TemplateId`.
- Add or extend Pydantic schema.
- Add template directory under `app/render/templates/{template_id}`.
- Add renderer dispatch in:
  - `create_render_zip()`
  - `create_render_zip_file()`
  - `_html_pages_for_request()`
- Add HTML page builder and row/component helpers.
- Add schema and rendering tests.
- Add sample fixture JSON.

Compatibility checklist:

- Keep old payload fields if existing UI/publications rely on them.
- If a grouped structure is introduced, decide whether to also send a flattened compatibility list.
- Requeue/sync code must reconstruct the same template id that was originally published.
- Preview page navigation should not be coupled to selected item state unless intentionally syncing to a selected row.
- Never assume today fixtures and weekly fixtures can share all layout constants.
